# Refactoring Guardrails (PR #1)

이 문서는 기능 추가 없이 구조를 정리할 때 지켜야 할 최소 안전장치입니다.

## 목표

- 기능 동작을 바꾸지 않고 유지보수성을 높인다.
- 변경 범위를 작게 나눠서 회귀 리스크를 줄인다.
- 데이터 흐름(생성/수정/저장)을 추적 가능하게 만든다.

## 기본 원칙

1. **한 PR = 한 목적**
   - PR 제목과 변경 이유가 1문장으로 설명 가능해야 한다.

2. **작은 변경 단위**
   - 파일 변경은 8개 이하를 원칙으로 한다.
   - 대량 포맷팅/정렬-only 변경은 금지한다.

3. **기능 변경 금지**
   - API 계약(필드/상태코드/에러문구)을 바꾸지 않는다.
   - DB 스키마 변경은 별도 PR로 분리한다.

4. **중복 제거 우선순위**
   - 같은 계산/검증 로직이 2회 이상 보이면 공통화 후보로 기록한다.

5. **데이터 흐름 가시화**
   - 변경한 파일 기준으로 “입력 → 처리 → 저장” 경로를 PR 본문에 명시한다.

## PR 본문 체크리스트

- 변경점 요약 (기능 변화 없음 명시)
- 테스트 방법(명령어)
- 리스크
- 롤백 방법

## 테스트 최소 기준

아래 중 하나를 만족해야 한다.

- 자동 테스트 3~5개 실행
- 또는 스모크 테스트(앱 실행 + 핵심 기능 1~2개 확인)

## 롤백 원칙

- 문제 발생 시 해당 PR 커밋을 통째로 되돌릴 수 있어야 한다.
- 리팩토링 PR에서 데이터 마이그레이션은 포함하지 않는다.
